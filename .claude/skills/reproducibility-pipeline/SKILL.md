# Reproducibility as day-1 architecture

The brief's contagion clause — *failure to meet the reproducibility requirement generates deductions in **all** rubric criteria* — makes this the only requirement that can erode every other point the team earns. Treat it as a design constraint from the first hour, not a packaging task for the last day. The test that matters: a judge clones the public repo on a clean machine, runs one command, and gets every figure and number in the report.

## Repository skeleton (create before any science)

```
repo/
├── README.md            # env setup + the one command that regenerates everything
├── requirements.txt     # frozen versions (pip freeze), not loose ranges
├── main.py              # THE single entry point (or one master notebook)
├── data/
│   └── grid_graph_8.json    # nodes, edges, weights, source URL, simplifications
├── src/                 # formulation, baselines, qaoa, plotting modules
├── results/
│   ├── raw/             # cached run outputs (json/csv) — committed
│   └── figures/         # regenerated by main.py — every report figure lands here
└── report/              # PDF + SDK statement
```

`python main.py` must regenerate **every figure and every number** cited in the report. If a figure exists that `main.py` cannot reproduce, either wire it in or drop it from the report — an orphan figure is a deduction waiting to be found.

## Determinism rules

- **One seed registry.** All randomness (numpy `default_rng`, optimizer initializations, GW hyperplane rounding, simulated annealing, shot simulation) flows from named, logged seeds — e.g. a `SEEDS` dict in one module. "≥5 runs with different initializations" means 5 *recorded* seeds, not 5 unseeded runs.
- **Freeze the environment.** `pip freeze > requirements.txt` from the actual working venv. Record the Python version in the README. Loose ranges (`qiskit>=1.0`) are how a judge's clone behaves differently from yours.
- **Version the data.** The graph file (with provenance and simplifications, per `grid-maxcut-modeling`) is committed, never generated ad hoc.

## Cache quantum runs — the emulator-specific pattern

Emulator/hardware jobs (Selene, H2 via Nexus) are slow, queued, or quota-limited, so `main.py` cannot re-execute them on a judge's machine. The standard resolution:

1. Every quantum run writes its **raw results** (counts, energies, parameters, seed, backend name, job ID, timestamp) to `results/raw/*.json` — committed to the repo.
2. `main.py` regenerates all figures **from the cached raw results** by default; a flag (`--rerun-quantum`) re-executes the jobs for those with access.
3. The README states this split explicitly. Figures-from-committed-raw-data is accepted reproducible practice; undocumented magic numbers are not.

Nexus job IDs double as evidence for the "ran on the official platform" points — record them.

## Clean-clone test (run it before submission, not after)

```
git clone <repo> /tmp/fresh && cd /tmp/fresh
python -m venv .venv && <activate> && pip install -r requirements.txt
python main.py            # must complete and fill results/figures/
```

Run this on day 2 the moment the pipeline first works end-to-end, then again before submission. Common failures it catches: imports of packages installed but missing from requirements.txt, absolute paths, files read from outside the repo, uncommitted data.

## Submission checklist (maps to the brief's deliverables)
- [ ] Public GitHub repo; README with env setup + single command.
- [ ] Frozen `requirements.txt`; Python version stated.
- [ ] `main.py` (or master notebook) regenerates every figure/number; quantum runs cached in `results/raw/` with job IDs.
- [ ] All seeds named and logged; graph data file versioned with provenance.
- [ ] Clean-clone test passed on the final commit.
- [ ] Report PDF (≤8 pages) and SDK statement (≤200 words) in the repo — see `report-pitch-deliverables`.

Pair with `maxcut-qubo-qaoa` (the pipeline being made reproducible) and route the final repo state through `quantum-code-audit` before submission.
